# Project Plan — Code Mastery


## Загальна інформація
**Назва проєкту:** Code Mastery  
**Тип проєкту:** Індивідуальна розробка (Full-Stack)  
**Розробник:** 1 особа  
**Технології:** React (Frontend), Node.js (Backend), MongoDB / PostgreSQL (БД)

**Мета:**  
Створити веб-платформу для навчання програмуванню, яка надає студентам доступ до інтерактивних курсів, а викладачам — інструменти для створення та проведення уроків.

---

## Управління ризиками

### Мета
- Виявлення ризиків, що впливають на терміни, якість або стабільність.  
- Визначення способів запобігання та реагування.  
- Регулярний перегляд стану ризиків після кожної ітерації.

### Матриця ризиків

| ID | Ризик | Ймовірність | Вплив | Тип | План реагування |
|----|--------|--------------|--------|------|----------------|
| R1 | Витік персональних даних через помилки у безпеці | Висока | Критичний | Технічний | Використати bcrypt, JWT, HTTPS; тестування валідації введення |
| R2 | Затримка виконання через навантаження на одну особу | Висока | Високий | Організаційний | Розбивати роботу на короткі спринти; визначати реалістичні цілі |
| R3 | Падіння продуктивності при великій кількості курсів | Середня | Високий | Технічний | Індексація БД, пагінація, кешування, оптимізація запитів |
| R4 | Відсутність код-рев’ю → архітектурні помилки | Середня | Середній | Якісний | Використовувати ESLint, Prettier, писати unit-тести |
| R5 | Некоректна адаптивність UI | Середня | Середній | Технічний | Тестування інтерфейсу на різних пристроях; Flex/Grid дизайн |

---

## Цілі управління проєктом
- Забезпечити контроль за термінами, якістю та ризиками.
- Організувати ефективне планування розробки однією особою.
- Забезпечити поступову реалізацію MVP через короткі ітерації.
- Мінімізувати технічні та часові ризики під час розробки.

---

## Організація проєкту
Оскільки над проєктом працює **один розробник**, усі ролі об’єднано:

| Роль | Відповідальність |
|------|------------------|
| **Project Manager** | Планування, моніторинг прогресу, управління ризиками |
| **Full-Stack Developer** | Архітектура, фронтенд, бекенд |
| **Tester** | Ручне тестування, unit-тести |
| **UI/UX Designer** | Макети інтерфейсу, адаптивність |
| **DevOps** | Контроль версій, деплой, середовище тестування |

## Методологія управління
**Підхід:** Спрощений Agile (індивідуальний Scrum).

- **Ітераційність:** короткі цикли по 1–2 тижні.  
- **Пріоритезація:** фокус на MVP.  
- **Планування:** ведення списку задач.  
- **Оцінка:** регулярне самооцінювання.  
- **Документація:** у форматі Markdown.  

---

## Планування виконання
- **Загальна тривалість:** 6 тижнів.  
- **Етапи виконання:**  
  1. Проєктування архітектури, налаштування середовища  
  2. Реалізація функцій користувача (авторизація, профіль)  
  3. Створення курсів, уроків і пошуку  
  4. Оцінка результатів  

- **Щоденна робота:**  
  - Визначення 2–3 ключових задач на день.  
  - Коміти у Git із короткими описами.  
  - Регулярна перевірка стабільності функціоналу.

---

## Управління якістю
- **Інструменти:** ESLint та Prettier.  
- **Методи перевірки:** ручне тестування після кожної фази.  
- **Критерії прийнятності:**  
  - Безпечне зберігання користувацьких даних;  
  - Відсутність критичних помилок у логах браузера;

- Зміни фіксуються безпосередньо у Git комітами з коротким описом.

- Вся проєктна документація зберігається в одному репозиторії.
